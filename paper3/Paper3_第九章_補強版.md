## 第九章：供應鏈威脅 —— 聚合中心投毒與上游憑證洩漏

> 本章節定義了**智能體攻擊鏈（Agentic Kill Chain, AKC）**中，關於「中間商與聚合中心」的投毒路徑，並擴展至「開發者自身的上游憑證洩漏」所形成的供應鏈斷點。

當開發者為了便利而使用第三方聚合工具（如 One API、New API 等）時，實則建立了一個具備「語義修改權」的單一崩潰點。然而，攻擊者不一定需要滲透聚合中心——他們可以從更上游的環節取得等效的進入點：**開發者自己洩漏的 API Key**。

---

### 9.1 三大攻擊向量（Attack Vectors）

#### 身份與權限偽造（Identity Forgery）—— Layer 1 供應鏈層

- **機制**：聚合中心具備攔截 API 金鑰的權限。
- **風險**：攻擊者可於背景以您的身份執行未授權的訓練、爬蟲任務，或將您的語義指紋轉賣。

---

##### 9.1.1 現實世界案例：GitHub 大規模 API Key 洩漏事件

> **研究者觀察（2026 年 5 月）**：GitHub 上正持續發生大規模的 API Key 洩漏事件，呈現出一個值得深入分析的攻擊模式。

**洩漏機制的技術路徑：**

```
開發者將 .env 或硬編碼 Key 提交至公開 Repo
        ↓
自動化爬蟲（如 truffleHog、gitleaks）在數分鐘內掃描並提取
        ↓
攻擊者以合法 API Key 以開發者身份發送請求
        ↓
AI 服務提供商的身份驗證層完全無法識別異常
        ↓
攻擊者取得「合法身份」，進入 Multi-Agent 信任鏈
```

**關鍵洞察：這不只是資安問題，而是信任鏈斷裂。**

在傳統資安框架中，API Key 洩漏被歸類為「憑證管理疏失」。但在 Multi-Agent 系統的語境下，其性質本質上不同：

| 傳統資安視角 | CLPMAI 視角 |
|------------|------------|
| Key 洩漏 = 帳號被盜 | Key 洩漏 = Agent 身份被偽冒 |
| 影響範圍：財務損失、資料外洩 | 影響範圍：整個 Agent 信任鏈崩潰 |
| 防禦手段：revoke key、輪替策略 | 防禦手段：需從架構層重新設計信任機制 |
| 事後可追蹤：billing 異常 | 事後難追蹤：語義注入無 forensic trace |

**最反直覺的發現：**

> 攻擊者不需要入侵任何伺服器、不需要破解任何加密、不需要繞過任何防火牆。
> 他們只需要等待開發者**自己**把金鑰推上 GitHub。

GitHub 的 Secret Scanning 功能雖然存在，但存在以下盲點：
1. **刪除 commit 不等於安全**：git history 永久保存，即使刪除檔案，爬蟲已在分鐘內取走
2. **私有 Repo 的錯誤信任**：開發者誤以為私有 Repo 安全，忽略協作者帳號被盜的風險
3. **Revoke 延遲窗口**：從洩漏到開發者察覺，平均存在數小時至數天的暴露窗口

---

#### 動態語義修改（Dynamic Semantic Modification）—— Layer 4 語義層 MITM

- **機制**：中間人攻擊，針對特定的「金融標的」或「決策指令」進行微調。
- **風險**：在不改動 JSON 結構的情況下，導致 AI 代理人產生認知漂移，觸發錯誤決策。

#### 緩存毒化（Cache Poisoning）

- **機制**：利用中心化的緩存機制進行集體操縱。
- **風險**：將錯誤、過期或經過修飾的資訊餵給不同使用者，引發群體性的認知偏差與市場操縱行為。

### 9.2 白帽研究員採證清單（Forensic Checklist）

| 測試項目 | 方法 | 指標 |
|------|------|------|
| **一致性對比** | 同時串接「原廠 API」與「聚合中心 API」，對比同一問題的 Token 分佈、數值精準度與語義偏移率 | 偏移率 > 5% 視為異常 |
| **延遲與流量分析** | 監控當請求包含敏感金融詞彙時的處理延遲 | 異常高延遲代表正在執行動態語義過濾 |
| **注入攻擊測試** | 使用 Layer 3（Token 層）攻擊向量，如注音、文言文、繁簡混合編碼 | 測試過濾器是否崩潰或產生預設誤轉譯 |

> **研究聲明**：本清單僅供防禦性研究與學術探討。在語義空間的「完美犯罪」成為現實前，我們必須先建立透明的採證標準。

### 9.3 上游洩漏：開發者自身的供應鏈斷點（Developer-Side Credential Breach）

> 9.1 描述的是「中間人攻擊」——攻擊者滲透聚合中心。本節描述更常見的攻擊路徑：攻擊者根本不需要滲透任何中間環節，因為**開發者自己就是最薄弱的供應鏈節點**。

#### 9.3.1 洩漏類型分類

| 類型 | 機制 | 現實頻率 |
|------|------|---------|
| **Git 提交洩漏** | .env 未加入 .gitignore，硬編碼 Key 推上公開 Repo | 極高——自動化爬蟲 7×24 掃描 |
| **日誌洩漏** | API Key 出現在 debug log、error message 中 | 高——開發者常忽略 log 輸出 |
| **聚合中心信任** | 使用第三方 proxy 服務（免費 API 聚合、低價轉發）時主動交出 Key | 高——台灣開發者社群普遍存在 |
| **協作環境洩漏** | Key 寫入 Jupyter Notebook、Google Colab，透過分享連結暴露 | 中——學習環境常見 |
| **前端硬編碼** | Key 直接寫入 JavaScript / HTML，瀏覽器 DevTools 可直接讀取 | 中——小型專案常見 |

#### 9.3.2 對 Multi-Agent 系統的特殊威脅

當 API Key 被盜取後，在單體應用中的影響是明確的（費用異常、配額耗盡）。但在 **Multi-Agent 架構**中，影響範圍升維：

```
被盜 Key 的攻擊者
        ↓
以合法 Orchestrator 身份發送指令
        ↓
下游 Agent 無法辨識指令來源是否為原始 Owner
        ↓
注入語義污染的指令（符合 Paper 1 的認知漂移攻擊）
        ↓
整個 Agent 信任鏈在不知情的情況下執行惡意邏輯
        ↓
無 forensic trace——所有 log 顯示「合法請求」
```

**這正是 CLPMAI 框架的核心危險性所在**：
攻擊在技術層面完全合法，防禦系統無從識別。

#### 9.3.3 與 Paper 2 Layer 0 的對應

Paper 2 的 Layer 0（Structural Prerequisites）定義了攻擊成立的結構性前提，其中「API key coupling」被列為信任假設的核心弱點。

本節提供了 Layer 0 的**現實世界驗證**：

> API key coupling 不只是理論上的結構弱點，而是今天正在 GitHub 上大規模發生的現實事件。Layer 0 的脆弱性已被現實證明。

### 9.4 複合攻擊鏈：GitHub 洩漏 × 認知層投毒（Compound Attack Chain）

> 本節提出一個將「憑證洩漏」與「認知層投毒」整合的複合攻擊場景，作為 Paper 3 的新型攻擊向量提案。

**攻擊名稱：Credential-Seeded Cognitive Poisoning（憑證植入型認知投毒）**

**擬提交 MITRE ATLAS ID：AML.TXX.010**

#### 攻擊流程：

```
Phase 1｜憑證獲取
GitHub 爬蟲掃描公開 Repo → 取得開發者 API Key
（攻擊成本：極低，自動化工具即可完成）

Phase 2｜身份偽冒
以洩漏 Key 的合法身份接入目標 Multi-Agent 系統
（防禦盲點：身份驗證層顯示「合法請求」）

Phase 3｜認知注入
參照 Paper 1/2 的 CLPMAI 攻擊框架，注入語義偏移指令
（攻擊核心：每條指令看起來都是正常的業務請求）

Phase 4｜累積漂移
多輪對話逐步推進認知漂移（Cognitive Drift）
（特徵：單一對話無異常，跨對話才出現偏移趨勢）

Phase 5｜觸發執行
Agent 在積累足夠的認知漂移後，自發執行攻擊者預設的偏誤決策
（Paper 3 對應病徵：NG-Loop、避重就輕、表演性坦誠）

Phase 6｜消除痕跡
所有 log 顯示正常業務流量，無任何 forensic trace
```

#### 為何現有防禦無效：

| 防禦機制 | 失效原因 |
|--------|---------|
| API Key 輪替 | 輪替週期內（通常數天至數週）窗口依然存在 |
| Rate Limiting | 攻擊者使用正常頻率，不觸發限流 |
| Anomaly Detection | 語義異常無法被流量分析偵測 |
| Multi-Agent 審計日誌 | 日誌顯示合法 Key 發出的合法請求 |
| LLM-as-a-Judge 自審 | 參照第五章 5.4「防禦性共謀」——多模型共同欺瞞 |

---

### 9.5 防禦建議（針對憑證洩漏向量）

#### 開發端（立即可執行）

1. **強制 pre-commit hook**：安裝 `gitleaks` 或 `detect-secrets`，在 commit 前自動掃描，Key 存在則阻斷 push
2. **.gitignore 標準化**：`.env`、`*.key`、`config/secret*` 列為強制排除項目，寫入專案 SKILL 規範
3. **Key 最小權限原則**：不同 Agent / 不同環境使用不同 Key，限制單一 Key 的操作範圍
4. **Revoke 自動化**：偵測到洩漏時（GitHub Secret Scanning 通知）立即自動 revoke，不依賴人工反應

#### 架構端（Multi-Agent 系統設計）

1. **請求來源驗證層**：在 Orchestrator 層加入除 API Key 之外的次要驗證（如 HMAC 簽章、IP 白名單）
2. **行為基線比對**：建立合法 Owner 的請求行為基線，偏離基線時觸發人工審核
3. **語義審計觸發器**：當指令語義出現第五章定義的「避重就輕」或「認知漂移」特徵時，強制中斷並回報

> **結論**：單靠技術手段（Key 管理、輪替）無法完全防禦此類攻擊。根本解決方案在於 **Multi-Agent 系統的信任架構重設計**——不能以 API Key 作為唯一的身份信任依據。

## 術語表補充（新增條目）

| 術語 | 英文 | 定義 |
|------|------|------|
| **憑證植入型認知投毒** | Credential-Seeded Cognitive Poisoning | 攻擊者透過洩漏的 API Key 取得合法身份，進而對 Multi-Agent 系統執行認知層投毒攻擊 |
