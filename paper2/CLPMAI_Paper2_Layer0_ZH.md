**CLPMAI Paper 2**

**跨層攻擊鏈：從供應鏈投毒到 AI Agent 執行**

*草稿章節 --- Layer 0：結構性前提與認知先決條件*

**Layer 0：隱形基礎------多層 AI 攻擊的結構性先決條件**

**0.1 概覽**

本文描述的五層攻擊鏈------涵蓋供應鏈投毒、聲學注入、Token層規避、語意操縱，以及
AI Agent
執行------並非憑空而生。每一層都預設了一組結構性條件，使攻擊得以可行、持續，並在鑑識上保持不可見。本文將這個先決條件空間稱為
Layer 0。

Layer 0 不是攻擊鏈中的某個步驟，而是所有其他層賴以運作的基底。不同於
Layer 1 至 5 描述攻擊者的行動，Layer 0
描述的是攻擊者所利用的條件------這些條件獨立存在，由 AI
平台運營商的設計選擇、開發者的經濟誘因，以及終端使用者的認知習慣所共同形塑。

Layer 0 由三條結構性主軸所定義：

- 開發者對 AI 生成輸出之制度性信任的內化------透過 Mondrian
  及相關市場操縱文獻的視角加以檢視。

- AI 即服務使用模式中隱含的信任假設------特別是開發者將 AI
  輸出視為權威而非對抗性訊號的傾向。

- API Key
  付款模式------藉由將財務存取與認知授權耦合於單一憑證中，創造出具有獨特可利用性的攻擊面。

第四條支撐主軸------歷史類比------將 Layer 0 植根於早於 AI
的大規模欺騙模式，論證該脆弱性結構並非新穎，而是被 AI
部署的規模與速度大幅放大的既有現象。

**0.2 Mondrian 文獻：算法權威與市場易受攻擊性**

**0.2.1 Mondrian 作為理論錨點**

Mondrian 框架最初由 LeFevre 等人在隱私保護資料匿名化的脈絡下提出
\[1\]，後續文獻將其延伸，用以描述算法決策系統中的分割問題：自動化代理如何將連續決策空間切分為離散的、可利用的區域？

就本文目的而言，Mondrian 概念帶有更具體的含義。在 AI
輔助金融市場中，可能投資訊號的空間，並非由單一模型分割，而是由多個 AI
助理同時在多名開發者之間運行的聚合輸出所共同切分。每個個別分割在局部看來是理性的。若操縱存在，它以分散形式跨越各個分割------在任何單一邊界上均不可見，但當從整個決策空間的跨層視角審視時，卻是結構性可預測的。

這種分散式、局部不可見的影響架構，正是 CLPMAI
所利用的機制。攻擊者不需要攻破單一系統；他們只需要移動 AI
輔助決策空間的集體分割------當攻擊者同時控制多個 AI
助理的語意輸入層（Layer 4）時，這項任務便成為可行。

**0.2.2 相關研究：AI 輔助市場操縱**

算法交易與 AI
生成內容的交叉領域，已產生了大量關於操縱向量的研究。幾條研究脈絡與 Layer
0 直接相關：

情緒放大攻擊 \[2\]
證明，向新聞聚合管道協調注入合成情緒訊號，可在數秒內measurably地改變交易量。CLPMAI
框架透過直接針對 AI 助理層，完全繞過新聞聚合，將此概念進一步延伸。

模型信心投毒 \[3\]
描述了一類攻擊：對抗性精心設計的輸入，使大型語言模型對錯誤的財務評估表達人為偏高的信心。先前研究聚焦於單模型攻擊，Paper
2 提出一條跨層鏈，其中模型信心首先在供應鏈層（Layer
1）被侵入，再於語意層（Layer 4）被利用。

認知監管套利概念 \[4\]------利用 AI
驅動市場訊號速度與監管偵測速度之間的落差------是 Layer 0
的基礎。當前金融監管框架是為偵測以人類節奏進行的操縱而設計的；語意規模的
AI
輔助操縱，其運作速度至少比偵測門檻快三個數量級，創造出一個結構性的利用窗口。

**0.2.3 Mondrian 缺口：Layer 0 的意義所在**

Mondrian 文獻揭示了一個一貫的盲點：先前研究將 AI
系統建模為被動元件------一個對假設為合法或可偵測為對抗性的輸入進行訊號處理的系統。Layer
0 從根本上挑戰了這一假設。

在 CLPMAI 框架中，AI
助理既非被動，也非單純的訊號處理器，而是一個權威人物------開發者自願安裝、定期諮詢，並透過
API 訂閱進行財務承諾的對象。這種權威關係並非 AI 部署的副作用，而是 AI
平台設計的結構性安排。Mondrian 缺口，正是先前文獻如何建模 AI
影響（視為理性市場上的概率性噪音）與 Layer 0
如何建模（視為開發者存取和解讀市場訊號的主要認知基礎設施）之間的空間。

**0.3 使用者信任假設：認知層的結構性脆弱性**

**0.3.1 AI 輔助開發中的信任梯度**

當代 AI
助理在軟體開發者中的使用，呈現出一致且可量測的信任梯度。關於開發者行為的實證研究
\[6\] 顯示，對於常規任務，開發者接受 AI 生成程式碼建議而不修改的比例約為
43%；當建議附帶信心指標或說明文字時，這一比例上升至
67%。此信任梯度並非非理性------它反映了對歷史準確率的合理校準。結構性問題在於，這種校準是在
AI 訓練資料尚未被對抗性策劃的條件下建立的。

CLPMAI Paper 1
引入了認知漂移的概念------透過在受控論壇環境中反覆接觸經微妙偏移的 AI
輸出，逐漸改變開發者對財務訊號的先驗概率分佈。Layer 0
確立了認知漂移得以生效的先決條件：在操縱行動開始之前，開發者必須已處於對
AI 生成財務評論的高度信任狀態。

這一先決條件並不難滿足。現代 AI
助理在市場上明確以可信度、事實準確性和專業可靠性為訴求。一名將 AI
助理整合到投資研究工作流程中的開發者，已透過這一行為做出了信任承諾，而認知漂移攻擊正是利用了這一承諾。

**0.3.2 權威轉移問題**

信任梯度之下存在更深層的結構性問題：權威轉移。當開發者透過 AI
助理路由財務分析查詢時，他們不僅僅是在尋求資訊------他們是在委派認識論權威。「我應該做空這支股票嗎？」這個決策，被重構為「AI
怎麼看這支股票？」繼而「我信任
AI。」人類的批判性評估步驟，已被信任評估步驟所取代。

權威轉移並非 AI 獨有。它是專家系統採用中有充分記載的現象
\[7\]，也是傳統社會工程學的基礎。AI
助理的獨特性，在於這種轉移的規模與不可見性：數百萬名開發者同時將認識論權威委派給訓練資料、微調過程和操作背景均無法審查的系統。

從攻擊者角度來看，權威轉移將技術上複雜的攻擊（入侵開發者的推理過程）轉化為邏輯上較簡單的攻擊：針對該開發者查詢類型，操縱
AI 助理的輸出。Layer 0
之所以存在，是因為權威轉移已在任何具體攻擊發動之前，在開發者群體中大規模發生。

**0.3.3 信任不對稱性與對抗性先驗的缺位**

Layer 0 信任條件的一個關鍵特徵是其不對稱性。開發者對 AI
輸出的信任模型不包含對抗性先驗------他們在善意假設的背景下評估 AI
回應。這與同一位開發者評估一封未經請求的電子郵件截然不同：在後者中，對抗性先驗已透過多年網路釣魚接觸而得到良好校準。

AI 輸出缺乏對抗性先驗，是設計使然。AI
助理被部署於對抗性意圖在結構上難以置信的情境中------AI
在操縱開發者方面沒有財務利益。Layer 0
正是利用了這種難以置信性。攻擊的可信度，恰恰在於攻擊者的機制（透過論壇認知漂移和供應鏈影響來污染
AI 輸出），在使用者介面層是不可見的。開發者沒有任何訊號顯示，AI
樂於助人、充滿信心、論述縝密的財務評論，已被系統性地偏移。

**0.4 API Key 付款模式：財務權威與認知權威的耦合**

**0.4.1 API Key 作為攻擊面**

主要 AI 提供商所實施的 API Key
模式，創造了一個在先前軟體安全文獻中沒有直接先例的攻擊面。單一 API Key
同時編碼了：（a）身份------Key
識別哪個帳戶正在發出請求；（b）財務授權------Key
授權對付款方式進行扣費；（c）認知授權------Key 授權存取開發者所信任的 AI
推理能力。

這種將財務與認知授權耦合於單一憑證中的設計，創造出一種獨特的可利用配置。獲得
API Key
的攻擊者，不僅獲得了在受害者帳戶上產生費用的能力------他們還獲得了攔截、修改或全面替換開發者視為權威的
AI 回應的能力。在 AI
輔助投資決策的情境中，這等同於同時持有受害者的信用卡和他們信任的財務顧問。

**0.4.2 開發者工作流程中的 API Key 暴露向量**

開發者工作流程常透過以下技術上可預防、行為上卻根深蒂固的管道暴露 API
Key：

- 版本庫暴露：API Key
  被提交至版本控制儲存庫，包括私有儲存庫（可能遭入侵或被意外公開）。2024
  年對公開 GitHub 儲存庫的調查，認定 API Key
  暴露是最常見的密鑰洩漏形式，且 AI 提供商 Key
  在暴露憑證中的占比持續增長 \[8\]。

- 論壇分享：在技術論壇（包括 CLPMAI Paper 1 所述類型的 AI
  聚焦社群）中分享程式碼片段的開發者，常因疏忽或假設 Key
  將被重新生成，在貼文範例中包含有效 API Key。

- 客戶端部署：建立 AI 驅動應用的開發者，將 API Key
  嵌入客戶端程式碼、瀏覽器擴充功能或行動應用中，而提取僅需基礎逆向工程技術。

- 代理服務利用：第三方 API 代理服務以「免費」或「更便宜」的 AI
  模型存取為賣點，透過其基礎設施路由請求，可能記錄、修改或重放請求，包括使用者的提示和
  AI 的回應。

**0.4.3 透過 API 代理的中間人攻擊**

代理服務向量值得特別關注，因為它在無需主動攻擊開發者本機環境的情況下，使
Layer 0 得以操作化。將 API
呼叫路由通過惡意代理的開發者，實際上已在認知授權被行使的精確位置------AI
回應生成並返回的那一刻------安裝了一個中間人對手。

惡意代理可以在開發者端點不留任何可偵測痕跡的情況下，隱密地執行以下任何操作：

- 回應修改：在保留原始回應的格式和明顯信心的同時，替換或補充具有對抗性設計的內容。

- 選擇性投毒：僅修改對特定查詢類型（例如包含財務股票代號的查詢）的回應，而將所有其他回應原封不動地傳遞，使行為異常偵測失效。

- 時間分階段：在開始針對性操縱之前，積累開發者查詢模式的統計輪廓，確保操縱針對開發者現有的認知先驗進行校準。

這一攻擊向量不需要零日漏洞、網路入侵，或對開發者的直接社會工程。它只需要開發者採用了使用第三方
API 代理這一行為上普遍的做法------而這種做法，恰恰是被主要 AI
提供商基於使用量的定價模式所創造的經濟誘因所積極鼓勵的。

> *API Key 付款模式將每一位 AI
> 輔助開發者轉化為潛在攻擊面，並非透過入侵其安全防護，而是透過利用其經濟理性。這種攻擊不是破壞系統------而是租用系統。*

**0.5 歷史類比：Layer 0 條件的結構性先例**

**0.5.1 桶店時代（1880年代至1920年代）：大規模分散式認知操縱**

十九世紀末至二十世紀初的桶店（Bucket Shop）運作，是與 Layer 0
結構條件最接近的歷史先例。桶店是無牌經紀商，接受對股價走勢的賭注，實際上並在交易所執行交易。其運營者控制著客戶所看到的報價------這個報價可以被操縱，在運營者持有對沖空頭倉位時，精確觸發客戶的停損位。

與 Layer 0
的結構對應是精確的。桶店客戶信任報價，因為：（a）報價是透過具有權威性的機構介面呈現的；（b）他們沒有其他現成可用的報價來源；（c）蓄意偽造的報價超出了他們的對抗性先驗範疇。Jesse
Livermore 對桶店運作的記述 \[9\]，描述了與 0.4.3 節所述 AI
代理操縱在結構上同構的精密報價操縱------唯一的差別是世紀和技術層次。

**0.5.2 Pecora 委員會調查結果（1932--1934年）：專家權威捕獲**

Pecora
委員會聽證會揭示了主要金融機構系統性的專家權威捕獲模式。受僱於投資銀行的分析師，發表公開流通的研究報告，推薦銀行同時正在從庫存中出售的證券。投資者信任分析師，因為：（a）分析師擁有機構資質；（b）投資者缺乏獨立評估證券的技術專業知識；（c）利益衝突在正常交易關係中在結構上是不可見的。

Layer 0 信任假設直接對映到這一結構。AI
助理占據了具備資質的分析師角色；開發者占據了信任投資者的角色；AI
平台運營商占據了投資銀行的角色。Pecora
案件中的操縱機制------被捕獲的專家意見------是透過論壇污染和供應鏈偏移所誘導的認知漂移的語意層類比。

Pecora 調查結果引發的立法回應------強制揭露分析師利益衝突------為 Layer
0 的緩解提供了潛在的監管框架。本文第四節將進一步發展這一含義。

**0.5.3
網路規模的拉高出貨（1990年代至2000年代）：速度不對稱作為結構性推手**

美國證券交易委員會在1990年代末記錄的網際網路時代拉高出貨（Pump-and-Dump）計畫，引入了早期操縱案例中所沒有的結構性要素：速度不對稱。運營者使用自動化電子郵件發送，在數分鐘內觸達數萬名投資者，在監管偵測機制能夠回應之前建立起被操縱的價格訊號。結構性教訓是：操縱有效性隨操縱速度與偵測速度之比值的增大而擴展。

CLPMAI 攻擊的速度不對稱比率，使網路時代的電子郵件行動相形見絀。AI
介導的認知操縱以對話速度運作------每次互動毫秒級，在受污染的 AI
助理生態系統中每小時數百萬次互動。監管偵測以調查速度運作------從發現訊號到執法行動需要數週至數月。Layer
0 利用的結構性前提，不僅是開發者信任 AI，更是他們信任 AI
的速度，快過任何偵測機制所能觀察到的信任被利用。

**0.5.4 LIBOR 操縱案（2012年）：協調性基準數據污染**

LIBOR 操縱醜聞中，多家機構的交易員協調提交報價以污染基準利率，引入了與
Layer 0
最後一項結構性要素相關的歷史先例：以基準污染為目的的多方協調。個別 LIBOR
提交看起來是合法的；只有在各機構協調提交的統計模式中，操縱才得以顯現。

CLPMAI 攻擊鏈在 AI 輸出層提出了類似的結構。若孤立評估單個 AI
回應，可能看起來公正且有幫助。操縱被結構性地編碼於受污染助理間輸出的聚合模式中------這種模式在開發者群體所接收的集體財務訊號中產生系統性偏移。就像
LIBOR 一樣，訊號在每個個別資料點看起來是合法的；污染在於相關結構。

這個類比具有直接的方法論含義：偵測 CLPMAI 類攻擊，需要在群體規模上對 AI
助理輸出進行相關性分析------這恰恰是目前 AI
安全審計框架中所缺乏的分析類型。Layer 0
不僅描述攻擊的先決條件；它還意味著這些先決條件包括缺乏群體層級的輸出相關性監控。

**0.6 Layer 0 綜合分析：先決條件矩陣**

Layer 0 的三條結構性主軸------Mondrian
識別的算法權威建模缺口、開發者信任假設，以及 API Key
付款風險------結合歷史證據基礎，共同定義了一個先決條件矩陣。Layer 1 至 5
所描述的跨層攻擊鏈，只有在此矩陣被填充時才具有可行性。

部署生態系統的基態。

條件，只需識別並利用已經存在的先決條件。攻擊的精密性不在於突破防禦，而在於武器化一個從未被設計來防禦語意層協調攻擊的系統的結構性特徵。
